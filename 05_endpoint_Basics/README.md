## エンドポイントをコードで見る

FastAPI では、
URL（パス）と処理の対応関係をコードで記述します。

以下は、もっともシンプルな例です。

```python
#main.py

from fastapi import FastAPI

app = FastAPI()

@app.get("/")
def read_root():
    return {"message": "Hello FastAPI"}
```

このコードで何が起きているか
1. FastAPI アプリケーションを作成しています
```
app = FastAPI()
```

この app が、
エンドポイントをまとめて管理する入口になります。

2. エンドポイントを定義しています
```
@app.get("/")
```

この1行で、

/ という URL に対して
HTTP の GET メソッドでアクセスされたとき
下に定義した関数を実行する

というルールを定義しています。

** ここで / が エンドポイントです。**

3. エンドポイントに対応する処理を書いています
```python
def read_root():
    return {"message": "Hello FastAPI"}
```


ブラウザやクライアントが / にアクセスすると

この関数が呼び出され

戻り値が HTTP レスポンスとして返されます

FastAPI では、
Python の辞書（dict）を返すと、自動的に JSON に変換されます。

## エンドポイントと HTTP の関係

同じ URL であっても、
HTTP メソッドが違えば別のエンドポイントになります。

例：
```python
@app.get("/items")
def get_items():
    return {"method": "GET"}

@app.post("/items")
def create_item():
    return {"method": "POST"}
```

この場合、

```
GET /items

POST /items
```

は 別のエンドポイントとして扱われます。

エンドポイントは「入口」である

ここで改めて整理します。

エンドポイントは
- 外部からサービスにアクセスするための入口です
-URL と HTTP メソッドの組み合わせで決まります

FastAPI では
デコレータ（@app.get など）によって明示的に定義します

エンドポイントの先にある処理やサービスの内容は、
この段階では気にしなくて構いません。

まずは、

** どの URL に、どんな入口が用意されているのか **

を意識することが大切です。
